118// ACK connectivity module lifecycle states. Besides being used to guide logic flows in this app, this
119// can be used to display info to the user, e.g. light a wifi icon when connected to Alexa, flash "setup"
120// when in setup mode, etc.
121typedef enum _ACKLifecycleState_t
122{
123 // Initial lifecycle when the HMCU application starts up. There's an immediate transition to UNAVAILABLE
124 // while the HMCU and ACK Connectivity Module finish booting and initializing.
125 ACK_LIFECYCLE_INITIAL,
126
127 // The ACK Connectivity Module is not available. This is the state after we've rebooted
128 // the ACK Connectivity Module (and at startup, since the first thing we do is reboot it).
129 // In this state we can (and must) still call ProcessEventIfAvailable, because transitioning
130 // out of this state requires us receiving the module-booted event. However we cannot successfully call
131 // ACK_GetStatus() in this state.
132 ACK_LIFECYCLE_UNAVAILABLE,
133
134 // The ACK Connectivity Module finished booting (or rebooting). This results in this code sending
135 // host configuration info to the ACK Connectivity Module, if it hasn't already been sent since entering
136 // this state.
137 ACK_LIFECYCLE_BOOTED,
138
139 // We successfully initiated a factory reset, which is in progress.
140 ACK_LIFECYCLE_FACTORY_RESET_IN_PROGRESS,
141
142 // The ACK Module is not registered but the setup process has timed out or is not running.
143 ACK_LIFECYCLE_NOT_REGISTERED,
144
145 // The ACK Module is in setup mode.
146 ACK_LIFECYCLE_IN_SETUP_MODE,
147
148 // This state can exist in a few scenarios:
149 // (1) The ACK Module is fully configured but not connected to Alexa, for example because wifi is down.
150 // (2) The device didn't finish a factory reset cycle before power was lost. In this case the user can
151 // try factory reset again.
152 ACK_LIFECYCLE_NOT_CONNECTED_TO_ALEXA,
153
154 // The ACK module is fully configured and is connected to Alexa.
155 ACK_LIFECYCLE_CONNECTED_TO_ALEXA
156}

201// ACK_CompleteDirectiveWith* routines are used to supply a response to a directive. You typically call these
202// from within an ACKUser_On* handler callback routine, in which case the directive response is sent before
203// ACK_Process returns to you. If your ACKUser_On* handler callback routine doesn't immediately know the
204// outcome of processing a directive, you can call an ACK_CompleteDirectiveWith* routine later, after
205// ACK_Process has returned. In that case, the directive response is sent immediately.
206// Every directive requires exactly one response. If you don't call an ACK_CompleteDirectiveWith* routine for
207// a given directive, Alexa will inform the end user of an error. Calling ACK_CompleteDirectiveWith* more
208// than once for a given directive will produce undefined behavior.
209
210// Tells Alexa that you completed processing a directive successfully, by sending a directive response
211// containing the properties identified by responsePropertiesBits. If changeReportPropertiesBits is non-0,
212// an Alexa change report is sent indicating that the properties designated in changeReportPropertiesBits
213// changed as a result of processing the directive. (For convenience a property can appear in both
214// responsePropertiesBits and changeReportPropertiesBits, but is effectively removed from
215// responsePropertiesBits for the purposes of creating the change report.)
